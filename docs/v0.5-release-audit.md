# v0.5 Release Audit

Scope: commits from `v0.4` to `HEAD` on local `main`.

This document records the release-quality review for v0.5. It is written from
the user's point of view: whether the feature is understandable, usable,
recoverable when the model/provider behaves unexpectedly, and safe to publish.

## Commit Groups

### Memory and Scratch

Commits:

- `307b399 Implement scratch and memory management actions`
- `9f8c852 Harden scratch note action coverage`

User-visible result:

- Durable memory can be inserted, updated, deleted, queried, and inspected with
  read-only SQL.
- Scratch notes support write/query/delete for temporary task checkpoints.
- Chat history search and deletion are separate from durable memory.

Quality judgment:

- Correct direction. It preserves Timem's core memory model instead of merging
  chat history, durable facts, and temporary work notes into one ambiguous
  store.
- The implementation keeps SQL read-only and routes writes through explicit
  actions, which is safer and easier to audit.
- Tests cover normal use, missing required fields, empty scratch query, delete
  misses, SQL write rejection, and chat history deletion safety.

### Provider Protocols and Prompt Cache

Commits:

- `a92f551 Add OpenAI Responses protocol support`
- `405e3f2 Keep bash approval policy out of prompt`
- `cb0babc Modularize incremental prompt cache planning`
- `919f5c3 Add structured output strategy`
- `e55c3ee Simplify env-based run examples`
- `10acc0f Prefer env-based command in help`
- `beebb95 Highlight overridden provider config`
- `d88f8ca Highlight model outside provider defaults`

User-visible result:

- `TIMEM_GATEWAY_PROVIDER` selects provider defaults.
- `TIMEM_API_PROTOCOL` selects wire format: `openai-compatible`,
  `openai-responses`, or `anthropic`.
- Static prompt and retained prompt deltas can be marked for provider prompt
  caching according to protocol support.
- README and help prefer sourcing an env file and then running `timem`.

Quality judgment:

- The provider/protocol split is necessary. A provider label is not the same as
  an API payload protocol.
- Cache strategy is now a module instead of ad hoc request formatting, which
  matches the design goal that prompt cache is a policy decision before provider
  packing.
- Tests cover protocol defaults, explicit base URL override, official OpenAI
  Responses shape, Anthropic cache-control blocks, OpenAI-compatible usage
  fields, and cached-token parsing.

### Shell Jobs, Bash Approval, and Local Work

Commits:

- `235806d Improve shell jobs and context shrink policy`
- `cf06688 Clarify write-capable Bash contract`
- `b64c9d0 Preserve Bash prompt update and restore rendering`
- `d4e8ee9 Require model-chosen wait for shell job status`

User-visible result:

- Bash approval mode is a runtime setting: `ask` or `approve`.
- Long-running shell work can run in background and be polled by `job_id`.
- The model must choose the wait timeout for `shell_job_status`; runtime does
  not guess.

Quality judgment:

- This matches the architectural principle that runtime should validate and
  execute structured actions, not interpret user intent semantically.
- Model-chosen wait reduces the "ask model again immediately" loop for long
  commands while keeping policy visible in the action protocol.
- Tests cover approval, denial, compound commands after approval, background job
  start/poll, timeout compatibility, and model-chosen wait.

### Prompt Delta, Slice, Scoring, and Shrink

Commits:

- `1ec46f9 Remove semantic durable score fallback`
- `c9c4cd8 Score submitted prompt deltas by sequence`
- `b1e1ee2 Cover durable score skipped-turn sequence`
- `10415cb Require durable scoring for prompt deltas`

User-visible result:

- Runtime-created prompt deltas start unscored.
- The model scores previously submitted deltas with `durable_ctx_score`.
- Runtime no longer infers score from natural-language keywords.
- Shrink reviews use input-token thresholds and exact delta/slice ids.

Quality judgment:

- Correct. Score timing is important: the model cannot score a new delta until
  it has seen that delta.
- Removing semantic fallback avoids hardcoded cases such as names, birthdays, or
  "remember" text in runtime.
- Tests cover scoring sequence, missing score repair, explicit delta ids,
  no semantic inference, shrink thresholds, 95% forced shrink, and
  slice/delta removal.

### Terminal UX and Interaction Reliability

Commits:

- `4872954 Show cancelled turn instead of curl failure`
- `4dd9233 Add modular thinking observation panel`
- `724bb49 Count concurrent observation transients`
- `01f2d43 Prefix observation lines without polluting responses`
- `fc273eb Make active observation lines visibly blink`
- `2d8d3c9 Use subtle color cycling for active observations`
- `c8b3c8e Allow continuing after round limit`
- `ba310f6 Clear wrapped submitted input lines`

- `8c4b837 feat(shell): collapse multiline paste in input`
- `ccf4d76 feat(shell): confirm edited paste placeholders`

User-visible result:

- Ctrl+C while thinking cancels the current turn instead of exiting or showing a

  action names such as `run_bash`.

Quality judgment:

- These changes are high user value because they address terminal friction that
  feels like brokenness even when the core is correct.
- Tests cover wrapped input clearing, CJK width, paste placeholder editing,
  keyboard prompts, Ctrl+C cancellation, observation scrolling, transient
  merging, and malformed model output not polluting the observation panel.
- Residual risk: true terminal behavior still depends on TTY state and should
  be smoke-tested manually before tagging.

### Runtime Config, Output Limits, and Stale Context

Commits:

- `65f187e feat(shell): add runtime token config`
- `1177fa4 feat(shell): confirm stale context resume`
- `8e05c4e fix(core): explain truncated model output`
- `db693f6 fix(shell): include provider error reasons`
- `a1f70fb fix(shell): split max token banner rows`
- `7f4f09a fix(shell): wrap config banner cells`
- `d434db3 docs(shell): align v0.5 runtime config names`

User-visible result:

- `TIMEM_MAX_LLM_INPUT` and `TIMEM_MAX_LLM_OUTPUT` are separate.
- `/config` can update runtime model/provider/protocol/base URL/token limits.
- If provider output is truncated by max output tokens, Timem explains that
  specific reason and can prompt to increase output budget and retry.
- After 3 hours idle with large context, Timem asks whether to continue old
  task context.
- Config banner wraps long cells and groups model/runtime/data fields.

Quality judgment:

- Splitting input/output tokens is correct. Input window controls context
  pressure; output token budget controls JSON truncation risk.
- Provider error reasons are safer than generic "HTTP 400/401" messages and
  better for user self-repair.
- Tests cover parse/help/config menu, default 100K/10K limits, output expansion
  prompt, provider error sanitization, stale context prompt threshold, and
  banner wrapping.

### Profiling and Audit

Commits:

- `600a5f0 Add runtime profiling command`
- `3a5a981 Improve profiling report readability`
- `06159ad Refine profiling token layout`
- `9061a90 Compact profiling token fields`
- `7ff123b feat(shell): add action audit log`
- `be10aaf fix(shell): keep memory outside audit dir`
- `8552329 test(shell): cover audit and observation edge cases`

User-visible result:

- `/prof` shows per-model calls, input/output/cached token counts, cache hit
  rate, wait time per 1K output tokens, local time, and storage sizes.
- Provider payload audit and action audit are separated under
  `data/<space>/audit/`.
- Runtime memory is stored under `data/<space>/memory/`, not under audit.
- Action audit groups actions by user turn and interaction.

Quality judgment:

- This improves supportability. When a long task fails or repeats, audit data
  can show what actions were requested and what the provider returned.
- Keeping memory outside audit fixes an important data-layout bug.
- Tests cover storage paths, action grouping, approval/denial audit, storage
  profile counts, and redaction.

## Robustness Coverage

Covered by tests:

- Malformed model JSON, fenced JSON, prose around JSON, invalid action shapes.
- Unescaped quotes and common JSON escape sequences.
- Provider truncation from OpenAI-compatible, OpenAI Responses, and Anthropic
  response shapes.
- Provider HTTP error bodies with unusual shapes, long text, and secret-looking
  strings.
- SQL injection/write attempts and invalid placeholder usage.
- Bash approval, denial, compound shell syntax, background jobs, and polling.
- Multi-line paste, edited paste placeholders, control-sequence stripping, CJK
  width, and wrapped input clearing.
- Observation panel scrolling, transient count, active color cycling, and
  internal action-name hiding.
- Prompt shrink thresholds, forced shrink, score timing, and no runtime
  semantic scoring.

Real provider smoke completed with `timem_shell/env`:

- Loaded the private env file with `source ./env`.
- Overrode runtime data to `/private/tmp/timem_v05_smoke` and space to
  `.v05_smoke` so release testing did not pollute normal memory.
- Ran one real provider call through the release binary with `--once-json`.
  Result: completed successfully, returned normal JSON, and reported provider
  token usage.
- Ran one real provider plus Bash action call with `TIMEM_BASH_APPROVAL=approve`
  and asked the model to execute `pwd`. Result: completed successfully with one
  `run_bash` action and a final answer containing the current directory.
- Inspected `action_audit.json`; it grouped both smoke turns and recorded the
  Bash action intent, input, status, and result summary.
- Scanned the smoke data for common secret markers; no API key or bearer token
  was found.

Remaining interactive terminal smoke before tag:

- Run `timem` in a real terminal with a real provider key.
- Ask two normal questions and confirm status line/cache display.
- Paste a multi-line prompt, move cursor, edit around the placeholder, submit.
- Run `/config`, change output token budget, then run `/prof`.

## Release Judgment

v0.5 is a strong release candidate. The automated test suite and real provider
smoke are broad enough for the core state machine, provider adapters, protocol
repair, memory/scratch tools, prompt shrink, audit, provider usage parsing, and
one Bash action loop.

Do not tag v0.5 until:

1. The remaining interactive terminal smoke passes.
2. `git status --short --branch` is clean.
3. The intended public remote receives the audited `main`.
4. The `v0.5` tag points at the same audited commit.
